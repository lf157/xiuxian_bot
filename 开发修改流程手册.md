# 开发修改流程手册

本文用于约束后续开发流程，保证“功能实现、可配置化、日志留痕”三件事同步完成。

## 1. 适用范围
- 新增功能。
- 现有功能改造（数值、规则、结算、文案、交互）。
- 运维/活动需要可调参数的功能。

## 2. 强制流程（必须按顺序）
1. 需求拆解
- 明确改动点：规则、数值、结算、展示、接口。
- 标记哪些参数需要运营可调。

2. 可配置化设计
- 所有“可能需要调整”的数值，优先放入 `config.json`。
- 代码中读取配置时，必须提供默认值与容错（类型转换失败回退默认值）。

3. 业务实现
- 服务层禁止新增硬编码关键参数。
- 关键流程（扣资源、发奖励、状态变更）优先保持原子性/幂等性。

4. Web 配置接入（必做）
- 在 `config.json` 增加配置项。
- 在 `web_local/static/i18n/config_fields_zh.json` 增加中文字段名。
- 如有前端/机器人文案涉及该值，必须改为动态读取，不可写死。

5. 文案一致性检查（必做）
- 检查“提示文案数值”与“实际结算数值”一致。
- 禁止出现“提示1小时，实际90分钟”这类漂移。

6. 基础校验
- JSON 文件语法校验通过。
- 受影响 Python 文件通过语法编译检查。
- 无测试环境时，至少完成静态自检与关键路径代码审查。

7. 日志留痕（必做）
- 每次改动完成后，更新 `CHANGELOG.md`。
- 记录必须放在最前面（倒序）。
- 每条记录必须包含：
  - 记录时间（`YYYY-MM-DD HH:MM (UTC+8)`）
  - 影响范围（文件/模块）
  - 修改摘要（做了什么、为何做）

## 3. 可配置项接入检查清单
- [ ] 该功能是否存在未来会调的阈值/比例/次数/消耗？
- [ ] 是否已写入 `config.json`？
- [ ] 是否已补充中文字段映射？
- [ ] 是否已移除展示层硬编码？
- [ ] 是否已验证文案和结算一致？
- [ ] 是否已写入 `CHANGELOG.md`？

## 4. 日志模板（复制即用）
```markdown
### [编号] 标题
- 记录时间：YYYY-MM-DD HH:MM (UTC+8)
- 影响范围：模块A、模块B、文件C。
- 修改摘要：
  - 要点1
  - 要点2
```

## 5. 命名约定
- 变更日志正式文件名：`CHANGELOG.md`。
- 历史临时日志文件（如 `LOCAL_BUGFIX_LOG.md`）不再使用。
